# HSTU KV-Cache AOTI End-to-End Build and Run Guide

## Purpose

This document describes the production workflow for building and running HSTU inference with KV-cache enabled AOTI export and C++ replay verification.

The workflow covers:

1. Building required custom operators and runtime libraries
2. Exporting the HSTU ranking model with KV-cache support through `torch.export` and AOTInductor
3. Starting the FlexKV-based KV-cache runtime service
4. Running the C++ replay executable against exported artifacts and dumped tensors
5. Running the Triton Server demo path for the exported KV-cache AOTI model

This guide is based on the checked-in workflow in [examples/hstu/inference/exported_with_kvcache_running_guide.sh](./exported_with_kvcache_running_guide.sh) and the container-oriented operational style in the local Triton/PyTorch guidebook.

## Scope

This guide is for the AOTI plus KV-cache path built around:

- `examples/hstu/inference/export_inference_gr_ranking_kvcache.py`
- `examples/hstu/inference/start_flexkv_server_for_kvcache_cpp.py`
- `examples/hstu/inference/cpp_inference/inference_hstu_gr_ranking_kvcache_exported_model.cpp`

It covers export, runtime setup, native C++ validation, and an optional Triton Server deployment and request-replay path for the same exported AOTI package.

## Expected Artifacts

At the end of the workflow, the following artifacts should exist:

1. Exported AOTI package in `examples/hstu/inference/hstu_gr_ranking_model/`
2. Replay tensors in `examples/hstu/inference/export_test_dump/`
3. C++ executable at `examples/hstu/inference/cpp_inference/build/inference_hstu_gr_ranking_kvcache_exported_model`
4. Runtime environment file generated by the KV-cache server starter, for example `/tmp/kvcache_cpp_runtime.env`
5. Optional Triton model repository content under `examples/hstu/triton_models/hstu_gr_ranking_kvcache/1/`
6. Optional Triton runtime libraries under `examples/hstu/triton_libs/`

## Environment Assumptions

This procedure assumes:

1. Linux with NVIDIA GPU access
2. CUDA toolkit available under `/usr/local/cuda`
3. Python and PyTorch already installed in the active environment
4. The repository root is available locally
5. The checkpoint directory and gin configuration are available

The Docker image used by this repository installs the following system dependencies for inference-related workflows:

- `liburing-dev`

The Docker image also installs Python-side build/runtime dependencies commonly needed by this workflow:

- `cython`

If you are not using the repository Docker image, ensure the equivalent dependencies are present.

## Build Overview

The KV-cache AOTI path depends on several separately built components:

1. DynamicEmb inference custom ops
2. HSTU custom ops from `examples/commons`
3. KV-cache manager custom ops
4. Python package install for `recsys_kvcache_manager`
5. C++ replay executable and runtime helper library under `examples/hstu/inference/cpp_inference`

## Step 1: Build DynamicEmb Inference Ops

Build `inference_emb_ops.so`:

```bash
cd ${REPO}/corelib/dynamicemb
mkdir -p torch_binding_build
cd torch_binding_build
cmake ..
make -j
```

Expected output:

```text
${REPO}/corelib/dynamicemb/torch_binding_build/inference_emb_ops.so
```

## Step 2: Build HSTU Custom Ops in `examples/commons`

Build the in-place extension and install the package:

```bash
cd ${REPO}/examples/commons
python setup.py build_ext --inplace
TORCH_CUDA_ARCH_LIST="8.0 8.6" python3 setup.py install
```

Use a different `TORCH_CUDA_ARCH_LIST` if your deployment targets a different GPU set.

## Step 3: Build KV-Cache Manager Custom Ops

Build the KV-cache manager C++ operator library:

```bash
cd ${REPO}/corelib/recsys_kvcache_manager/build
cmake ..
make -j 4
```

This produces the shared library consumed by the runtime and C++ replay flow.

## Step 4: Install the `recsys_kvcache_manager` Python Package

The export pipeline imports the Python package, not only the shared library. Install it explicitly:

```bash
cd ${REPO}/corelib/recsys_kvcache_manager
TORCH_CUDA_ARCH_LIST="8.0 8.6" pip3 install --no-build-isolation .
```

If your environment already has a matching installed build, you can skip reinstalling. For reproducibility, explicit installation is recommended.

## Step 5: Build the C++ KV-Cache Replay Executable

Build the C++ demo without the wrapper script:

```bash
export PATH=/usr/local/cuda/bin:${PATH}
export CMAKE_PREFIX_PATH="$(python3 -c 'import os, torch; print(os.path.join(os.path.dirname(torch.__file__), "share", "cmake"))')"

cmake -S "${HSTU_DIR}/inference/cpp_inference" -B "${HSTU_DIR}/inference/cpp_inference/build"
cmake --build "${HSTU_DIR}/inference/cpp_inference/build" --target inference_hstu_gr_ranking_kvcache_exported_model -j 8
```

Expected outputs:

1. `examples/hstu/inference/cpp_inference/build/inference_hstu_gr_ranking_kvcache_exported_model`
2. `examples/hstu/inference/cpp_inference/build/libhstu_cuda_ops_runtime.so`

## Step 6: Run Python Export for KV-Cache AOTI

Run the export workflow from the HSTU directory:

```bash
cd ${HSTU_DIR}
python3 inference/export_inference_gr_ranking_kvcache.py \
  --gin_config_file ${GIN} \
  --checkpoint_dir ${CKPT} \
  --max_bs 2
```

This step performs all of the following:

1. Builds the exportable model wrapper for KV-cache inference
2. Exports the model through `torch.export`
3. Produces the packaged AOTI archive under `inference/hstu_gr_ranking_model/`
4. Produces replay tensors under `inference/export_test_dump/`

### Output Contract

The exported wrapper returns:

1. `logits` as `float32` on CPU
2. `task_ids` as `int64` on CPU

This is the contract consumed by the C++ replay executable.

## Step 7: Start the FlexKV Runtime Service

The C++ replay executable depends on a runtime environment file created by the KV-cache server starter.

Start the service as follows:

```bash
cd ${HSTU_DIR}
rm -f ${ENV_FILE}
python3 inference/start_flexkv_server_for_kvcache_cpp.py --env_file ${ENV_FILE} > kv.log 2>&1 &
```

Then verify that the environment file was created:

```bash
cat ${ENV_FILE}
```

Load the runtime variables into the current shell:

```bash
source ${ENV_FILE}
```

### Important Note

Do not rely on `setup_kvcache_cpp_runtime_env.sh` for this workflow. That script is referenced in an informal guide, but it is not present as a checked-in file in this repository snapshot. The supported path is to let `start_flexkv_server_for_kvcache_cpp.py` generate the env file directly.

## Step 8: Run the C++ Replay Verification

Run the replay executable against the exported package and dumped tensors:

```bash
${HSTU_DIR}/inference/cpp_inference/build/inference_hstu_gr_ranking_kvcache_exported_model \
  ${HSTU_DIR}/inference/hstu_gr_ranking_model \
  ${HSTU_DIR}/inference/export_test_dump
```

The executable will:

1. Load the packaged AOTI model
2. Load NVE metadata and weights
3. Replay each dumped batch
4. Compare C++ outputs against exported reference tensors
5. Report `max_abs_diff` for each batch

## Step 9: Triton Server Demo for the KV-Cache AOTI Model

This section extends the native replay flow into Triton serving.

The Triton path has two major parts:

1. Build or install a PyTorch backend with the required PT2/AOTI behavior
2. Launch Triton with the exported model, custom operator libraries, and FlexKV runtime

## Operational Notes

### Library Loading

The C++ replay executable expects the following custom/runtime libraries to be buildable or loadable:

1. DynamicEmb inference ops
2. HSTU runtime ops from the C++ demo build
3. Paged KV-cache ops from `examples/commons`
4. KV-cache manager ops from `corelib/recsys_kvcache_manager`
5. FBGEMM shared libraries

If replay fails during library loading, validate the earlier build steps first.

### Runtime Environment Variables

The KV-cache runtime requires environment variables such as:

1. `SERVER_RECV_PORT`
2. `GPU_REGISTER_PORT`
3. `KVCACHE_MANAGER_NUM_LAYERS`
4. `KVCACHE_MANAGER_NUM_KV_HEADS`
5. `KVCACHE_MANAGER_HEAD_SIZE`
6. `KVCACHE_MANAGER_TOKENS_PER_PAGE`
7. `KVCACHE_MANAGER_MAX_BATCH_SIZE`
8. `KVCACHE_MANAGER_MAX_SEQUENCE_LENGTH`
9. `KVCACHE_MANAGER_DTYPE`

These are emitted into `${ENV_FILE}` by `start_flexkv_server_for_kvcache_cpp.py`.

The same environment contract is also required by the Triton demo path when the model is loaded inside Triton.

### Build and Runtime Separation

Treat build and runtime as separate stages:

1. Build all shared libraries and Python packages first
2. Export the AOTI model and replay tensors second
3. Start the FlexKV runtime third
4. Run C++ replay last

This avoids mixing build failures with runtime failures.

For the Triton path, add a fifth stage:

5. Stage the exported model and libraries into the Triton runtime and only then start `tritonserver`

## Failure Triage

If the workflow fails, narrow the problem by stage:

1. Build failure in `examples/commons`: validate PyTorch headers, CUDA toolkit, and `TORCH_CUDA_ARCH_LIST`
2. Build failure in `recsys_kvcache_manager`: validate CMake configuration and compiler toolchain
3. Export failure: validate the gin file, checkpoint path, and Python package installs
4. Runtime env failure: validate `${ENV_FILE}` generation and required variables
